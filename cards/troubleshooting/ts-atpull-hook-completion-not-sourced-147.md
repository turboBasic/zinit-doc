## Summary

When a completion file is generated by an `atpull` hook (e.g. `kubectl completion zsh > _kubectl`), zinit reports "No files for compilation found" on update and the generated completion is not picked up.

## Symptom

```
[update] Continuing with the update because run-atpull'' ice given.
No files for compilation found.
Warning: ∞zinit-compile-plugin-hook hook returned with 1
```

The completion file `_kubectl` is generated but not linked into zinit's completions directory.

## Cause

Zinit's completion detection runs before the `atpull` hook generates the file. On update, the file detection does not re-run after hooks complete, so new completion files created during `atpull` are not registered.

## Fix / Workaround

No fix available as of issue filing; track https://github.com/zdharma-continuum/zinit/issues/147

Workaround: use `nocompile` ice to prevent compile errors, and explicitly reinstall completions after update via `creinstall`:

```zsh
zinit lucid wait has"kubectl" for \
  id-as"kubectl_completion" as"completion" \
  atclone"kubectl completion zsh > _kubectl" \
  atpull"%atclone" run-atpull nocompile zdharma-continuum/null
```

After updating, run: `zinit creinstall kubectl_completion`

Or add `atload"zinit creinstall %id"` to trigger reinstall automatically.
